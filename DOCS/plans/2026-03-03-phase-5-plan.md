**"The grid speaks about itself"**

### What It Does
Pure function: takes a grid + frame index → returns a structured natural language description. No ML, no DOM, no network. This is the foundation for everything else in Phase 5 — the description feeds Gemini prompts (5.3), populates project.ai_context, and is the text representation of the visual medium.

### API

```js
/**
 * describeGrid(grid, frameIndex?, options?) → GridDescription
 *
 * @param {object} grid - A valid .grid object
 * @param {number} frameIndex - Frame to describe (default 0)
 * @param {object} options
 * @param {string} options.detail - 'brief' | 'standard' | 'full'
 * @param {boolean} options.includeAudio - Include audio channel summary
 * @param {boolean} options.includeSpatial - Include spatial channel summary
 * @returns {GridDescription}
 */

// GridDescription shape:
{
  summary: "An 86×118 dense grid with 10,142 cells...",
  composition: {
    dimensions: "86 columns × 118 rows",
    cellCount: 10142,
    fillRatio: 0.997,      // cells / (width × height)
    densityAvg: 0.72,
    densityDistribution: "heavy" // "sparse" | "moderate" | "heavy" | "saturated"
  },
  palette: {
    dominant: "#192123",     // most frequent color
    accent: ["#00ff88", "#ff4444"],  // top 3 non-dominant
    colorCount: 847,
    warmth: "cool"           // "warm" | "cool" | "neutral" | "mixed"
  },
  semantics: {
    solid: 0.72,             // fractional breakdown
    fluid: 0.15,
    void: 0.08,
    emissive: 0.05,
    distribution: "solid-dominant with fluid veins"
  },
  regions: [
    { id: 0, label: "top-center", semantic: "emissive", area: 142,
      description: "Bright cluster at top center — likely text or title" },
    { id: 1, label: "center", semantic: "solid", area: 4200,
      description: "Large dense mass at center — primary subject" }
  ],
  audio: {                   // if includeAudio
    noteRange: "C2–G6",
    avgVelocity: 91,
    channelDistribution: { lead: 0.4, bass: 0.2, pad: 0.3, arp: 0.1 }
  },
  spatial: {                 // if includeSpatial
    avgHeight: 0.72,
    peakRegion: "center",
    materialBreakdown: { solid: 0.72, fluid: 0.15, void: 0.08, emissive: 0.05 }
  },
  prompt: "A cool-toned dense composition on dark background. Bright emissive text at top. Large solid central mass with fluid veins. Accent colors: green, red."
}
```

### Key Algorithm: Region Detection

- Simple connected-component labeling on the semantic map:
```
1. Build semantic grid (W×H) via getSemanticMap()
2. Flood-fill to label connected regions of same semantic type
3. For each region: compute centroid, bounding box, area, label with
   spatial term (top-left, center, bottom-right, etc.)
4. Sort by area descending, take top N (configurable, default 8)
5. Generate natural language per region
```

- This is the same algorithm used in image segmentation but operating on the semantic channel, which is already categorical. No ML needed — just flood fill.

### Key Algorithm: Palette Extraction

```
1. Collect all cell colors from frame
2. Frequency count → sort descending
3. Dominant = most frequent
4. Cluster remaining by hue distance (simple HSL grouping, 30° buckets)
5. Accent = top 3 cluster centroids excluding dominant
6. Warmth = weighted average hue: <60° or >300° = warm, 120-240° = cool
```

### Key Algorithm: Prompt Synthesis
- The prompt field concatenates: warmth + density descriptor + background + region descriptions + accent colors. This becomes the seed for Gemini (5.3) or local SLM (5.4b).

- Template
```
"A {warmth}-toned {densityDistribution} composition on {dominant} background. {region[0].description}. {region[1].description}. Accent colors: {accent.join(', ')}."
```

## Task 5.4a — Text-to-Grid Generator (Template Engine)

**"Describe a place, get a grid"**

### What It Does
Pure function: takes a text description → returns a .grid object. No ML — this is a keyword-driven template engine that dispatches to existing generators and a new layout engine. The "AI" is structural: it parses intent from text and composes grid elements.

This is where `Doxascope` "My verse, another project" worldbuilding happens: "A dense city seen from above, narrow alleys, central plaza with fountain, walls on north side" → populated .grid.

### API

```js
/**
 * textToGrid(prompt, options?) → { grid, interpretation }
 *
 * @param {string} prompt - Natural language description
 * @param {object} options
 * @param {number} options.width - Grid width (default 40)
 * @param {number} options.height - Grid height (default 20)
```

### Keyword Vocabulary (shipped with the module)

```js
const VOCABULARY = {
  // Terrain keywords → generators + semantics
  terrain:  { triggers: ['landscape', 'terrain', 'mountain', 'hill', 'valley'],
              generator: 'terrain', semantic: 'solid' },
  water:    { triggers: ['water', 'ocean', 'sea', 'river', 'lake', 'fluid', 'mist'],
              generator: 'wave', semantic: 'fluid' },
  city:     { triggers: ['city', 'town', 'building', 'wall', 'alley', 'street'],
              generator: 'geometric', semantic: 'solid' },
  sky:      { triggers: ['sky', 'stars', 'night', 'space', 'void', 'cosmos'],
              generator: 'rain', semantic: 'void' },
  energy:   { triggers: ['energy', 'pulse', 'glow', 'emissive', 'light', 'fire'],
              generator: 'pulse', semantic: 'emissive' },
  pattern:  { triggers: ['pattern', 'mandala', 'spiral', 'fractal', 'radial'],
              generator: 'mandala', semantic: 'solid' },
  chaos:    { triggers: ['chaos', 'noise', 'random', 'scatter', 'matrix'],
              generator: 'noise', semantic: 'solid' },
  gradient: { triggers: ['gradient', 'fade', 'transition', 'horizon'],
              generator: 'gradient', semantic: 'solid' },

  // Doxascope-specific (future-proofing)
  mist:     { triggers: ['mist', 'coherence', 'dissolution', 'fracture'],
              generator: 'noise', semantic: 'fluid', colorHint: '#667788' },
  door:     { triggers: ['door', 'portal', 'transcendence', 'transformation', 'annihilation'],
              generator: 'pulse', semantic: 'emissive', colorHint: '#d4af37' },

  // Layout modifiers (not generators — affect zone placement)
  modifiers: {
    position: ['top', 'bottom', 'left', 'right', 'center', 'north', 'south', 'east', 'west'],
    density:  ['dense', 'sparse', 'thick', 'thin', 'heavy', 'light'],
    size:     ['large', 'small', 'wide', 'narrow', 'tall', 'short']
  }
};
```

### Parsing Pipeline
```
1. Tokenize prompt → lowercase words
2. Match tokens against VOCABULARY triggers → collect zone intents
3. Match modifier tokens → assign to nearest zone intent
4. If no zones matched → fallback: whole-grid 'noise' generator
5. Layout engine: divide grid into zones based on position modifiers
6. For each zone: call appropriate generator with zone bounds + seed
7. Compose zones into single frame
8. Return { grid, interpretation } where interpretation shows the parse
```

### Layout Engine (Zone Allocation)
```
Grid divided into 3×3 sectors:
  ┌─────┬─────┬─────┐
  │ NW  │  N  │ NE  │
  ├─────┼─────┼─────┤
  │  W  │  C  │  E  │
  ├─────┼─────┼─────┤
  │ SW  │  S  │ SE  │
  └─────┴─────┴─────┘

"walls on north side" → N sector: geometric generator
"central plaza" → C sector: pulse generator
"narrow alleys" → fill remaining sectors: rain generator, low density
```

## Task 5.2 — Image Upscale Pipeline (Tier 1)

**"The grid renders in HD"**

### What It Does

- What It Does: Takes the current grid canvas render → passes through an ONNX image upscaling model → produces a 2x or 4x resolution image. Client-side, no server. The ASCII art becomes a crisp HD image suitable for print or publication.

### Files
- Create: `src/consumers/ai/upscaler.js`
- Create: `tests/test-upscaler.js`

### CDN Loading Pattern (matches Three.js)

```js
// In head.html — conditional ONNX Runtime Web load
(function () {
  const s = document.createElement('script');
  s.src = 'https://cdn.jsdelivr.net/npm/onnxruntime-web/dist/ort.min.js';
  s.onload = () => { window._ortLoaded = true; };
  s.onerror = () => { window._ortLoaded = false; };
  document.head.appendChild(s);
})();
```

### API

``` js
/**
 * isUpscaleAvailable() → boolean
 * upscaleDefaults() → { scale: 2, modelUrl: '...', tileSize: 256 }
 * upscaleCanvas(canvas, options?) → Promise<{ok, blob?, error?}>
 *
 * Pipeline:
 *   canvas → ImageData → Float32 tensor → ONNX session.run()
 *     → Float32 output → clamp/quantize → ImageData → canvas → Blob
 *
 * Tiling: Large images split into overlapping tiles (tileSize × tileSize)
 * to stay within GPU memory. Tiles reassembled with seam blending.
 */
```

### Model Selection

**Needs web search**: Best ONNX-compatible image upscaler that runs in browser, <50MB, 2x or 4x scale. Candidates: Real-ESRGAN (ONNX export exists), Swin2SR, or compact alternatives.

> The model file is loaded on first use (lazy), cached in OPFS for subsequent runs. Not bundled — fetched from CDN or user-provided URL.

## Task 5.5 — Circuit Breaker

**"The grid protects itself"**

### What It Does

- Tracks API usage, automatically falls back when quotas are hit. Prevents surprise bills and handles the free→paid tier boundary.

### Files

- Files

- Create: `src/consumers/ai/circuit-breaker.js`
- Create: `tests/test-circuit-breaker.js`
- Modify: `src/consumers/ai/gemini-bridge.js` (wrap calls through breaker)
- Modify: `build.js`, `tests/run-all.js`

### API

```js
/**
 * createCircuitBreaker(options?) → CircuitBreaker
 *
 * options:
 *   maxRequestsPerMinute: 15    (Gemini free tier)
 *   maxRequestsPerDay: 1500     (Gemini free tier)
 *   cooldownMs: 60000           (pause after trip)
 *   storageKey: 'grid_cb'       (localStorage key)
 *
 * Methods:
 *   canRequest() → boolean
 *   recordRequest() → void
 *   recordFailure(errorType) → void
 *   getStatus() → { state, requestsToday, requestsThisMinute, nextResetMs }
 *   reset() → void
 *
 * States: 'closed' (normal) | 'open' (tripped) | 'half-open' (testing)
 *
 * Storage: usage counters in localStorage, survives page reload
 * Day boundary: midnight UTC reset
 */
```

**COnsider:** Models adapters for OpenAI, Anthropic, OLLama, etc. 

### State Machine
```
CLOSED (normal)
  ├── canRequest() → true
  ├── On 429/quota error → trip → OPEN
  └── On daily limit reached → trip → OPEN

OPEN (tripped)
  ├── canRequest() → false
  ├── UI shows: "Tier 2 paused — using local AI"
  └── After cooldownMs → HALF-OPEN

HALF-OPEN (testing)
  ├── canRequest() → true (one request)
  ├── On success → CLOSED
  └── On failure → OPEN (restart cooldown)
```

### Tests (~25 tests)

```
— initial state: closed, canRequest true                        (2)
— recordRequest increments counter                              (2)
— rate limit: RPM exceeded → open                               (3)
— rate limit: RPD exceeded → open                               (3)
— open state: canRequest false                                  (2)
— cooldown: after elapsed time → half-open                      (2)
— half-open: success → closed                                   (2)
— half-open: failure → open again                               (2)
— recordFailure with 'rate_limit' → trip immediately            (2)
— reset clears all counters                                     (1)
— getStatus returns correct shape                               (2)
— localStorage persistence: reload preserves counters           (2)
— total                                                         (~25)
```

### Exit Criteria

- Free tier exhaustion → automatic fallback to Tier 1
- Cooldown period → auto-retry
- Usage visible in UI (requests today, state)
- localStorage survives page reload

## Decisions

### 1. Description engine is pure templates, not ML

- 5.1 uses algorithmic analysis (flood fill, color clustering, heuristics) — not a language model. This keeps it Tier 0, deterministic, testable, and fast. An SLM could improve the prose quality, but the structured data is what consumers need.

### 2. Text-to-grid is keyword dispatch, not NLP
- 5.4a parses keywords against a vocabulary table and dispatches to existing generators. This is intentionally "dumb AI" — the intelligence is in the layout engine, not the parser. A local SLM upgrade path exists (parse prompt with SLM → extract structured intents → same layout engine) but is deferred to Phase 8.

### 3. CDN-only for ML deps, no npm install

- Transformers.js and ONNX Runtime Web load via CDN script injection, same pattern as Three.js. This preserves `"dependencies": {}` in package.json. The tradeoff: first load is slower (model download). Mitigation: OPFS caching of model files.

### 4. Model files are user-downloaded, not bundled

- The upscaler model (~20-50MB) and segmentation model (~50-100MB) are fetched on first use and cached in OPFS. They are NOT part of the build. This keeps `dist/index.html` at ~225KB.

### 5. Gemini key in localStorage, never in .grid

- API keys never enter the .grid format. `project.ai_context` stores descriptions and prompts (the output of AI), never credentials.

### 6. Smart import wraps, never replaces

- `smartImageToGrid()` calls `imageToGrid()` internally and enriches the result. If ML fails at any point, the output is identical to Task 1.6. No regression possible.

----

## Deferred

### SLM Text Generation (local 0.5B model for prose)

- Reason: The template engine in 5.1 + 5.4a covers the structured use case. An SLM would improve natural language quality but adds 200-500MB model download and significant complexity. Revisit at Phase 8 (Studio) when the AI panel gets a conversational interface.

### Veo Video Generation

- Reason: Veo API access may be limited/waitlisted. The stub exists in gemini-bridge.js but the UI button stays disabled until verified. The .grid → MP4 pipeline (Phase 6) is the offline path.

### AI-assisted grid editing (copilot mode)

- Reason: "Paint with AI assistance" (suggest next cell, auto-complete patterns) requires real-time SLM inference at <100ms latency. This is Phase 8 territory — after the studio UI exists.
